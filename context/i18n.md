# Domain: Internationalization (i18n)

## Overview
Multi-language support via `i18next` + `react-i18next`. Translation resources can be loaded statically or fetched from a remote endpoint (Tolgee-compatible). Language is set via `ChatConfig.locale`.

## Key Files
| File | Purpose |
|------|---------|
| `src/i18n/` | i18n setup, config, and translation resources |
| `src/index.ts` | Exports `TranslationProvider`, `useTranslations`, `fetchTranslations` |

## Public API
```ts
// Wrap app in provider — required before any translated component renders
<TranslationProvider locale="en" apiBase="...">
  <ChatWrapper ... />
</TranslationProvider>

// Hook for translating strings in custom components
const t = useTranslations();
t('chat.placeholder')  // → "Type a message..."

// Fetch translations at build time or SSR
const translations = await fetchTranslations({ locale: 'en', apiBase: '...' });
```

## Translation Key Namespaces
- `chat.*` — input, placeholders, send button labels
- `error.*` — error messages shown to users
- `tool.*` — tool execution labels and status text
- `connection.*` — connection status notifications
- `reasoning.*` — reasoning/thinking section labels

## Adding a New Translation Key
1. Add the key + English value to the default resource bundle in `src/i18n/`
2. Add the key to all other locale files
3. Use `t('your.key')` in the component
4. Push to Tolgee via `npx dotenv -e .env.local -- tolgee push` if remote sync is configured

## Locale Detection Priority
1. Explicit `locale` prop on `TranslationProvider`
2. `ChatConfig.locale`
3. Browser `navigator.language`
4. Fallback: `en`

## Remote Translation Sync (Tolgee)
- `TOLGEE_API_KEY` and `TOLGEE_PROJECT_ID` in `.env.local` enable push/pull
- `fetchTranslations()` fetches from remote on demand — useful for runtime locale switching

## Gotchas
- Missing translation keys silently fall back to the key string (e.g., `"chat.placeholder"`) — watch for these in UI reviews
- `TranslationProvider` must be mounted before any `useTranslations` call — don't use outside the provider tree
- `react-i18next` v16 uses a different `useTranslation` hook signature than v15 — check import if upgrading
- Language switching at runtime works, but causes a full re-render of the translation provider subtree
